# E2E Flow + Agent Impersonation + Browser Integration Security Audit

## Executive Summary

The broker implements a multi-stage task lifecycle with basic authorization checks, but has critical security gaps in agent identity verification and GitHub token management. Self-acceptance checks exist but rely on address comparison only. GitHub tokens are stored in-memory with no encryption or persistence validation.

## E2E Flow Map

```
POST /ideas → createIdea() → idea.status='unfunded'
     ↓
POST /ideas/:id/plan → generateBrief() → brief created, jobs.status='created'
     ↓
POST /jobs/:id/claim → claimJob() → job.status='claimed', claim record created
     ↓
POST /jobs/:id/submit → submitJob() → job.status='submitted' OR 'rework' (based on scoring)
     ↓
POST /jobs/:id/accept → acceptJob() → job.status='accepted', payout released
     ↓
POST /jobs/:id/reject → rejectJob() → job.status='rework', credits refunded
```

## Findings

| Severity | Location | Description | Recommendation |
|----------|----------|-------------|----------------|
| **HIGH** | jobService.ts:206-275 (submitJob) | Agent identity not cryptographically verified at submission. The `agentFingerprint` parameter is trusted without verification against the `accountAddress`. An attacker can pass any fingerprint. | Verify `agentFingerprint` matches `computeAgentFingerprint()` result for the claimed agent type/version before accepting submission. Add signature verification. |
| **HIGH** | githubService.ts:158-186 | GitHub tokens stored in plaintext in-memory Map with no encryption. Tokens are exposed in process memory and can be dumped if the process is compromised. | Encrypt tokens at rest using AES-256-GCM with key from env var. Use Redis with TLS for production instead of in-memory Map. |
| **MEDIUM** | githubService.ts:55-84 | No SSRF protection on GitHub API calls. The `fullName` parameter in `getRepoContext()` is passed directly to API URL without validation. | Validate `fullName` matches GitHub repo name regex before API call. Add allowlist for github.com domain only. |
| **MEDIUM** | githubService.ts:63 | No scope validation on GitHub token. Token could have broader scopes than needed (e.g., delete_repo, admin:org). | Request minimal scopes (public_repo, repo:status) during OAuth flow. Validate token scopes before use. |
| **LOW** | jobService.ts:287-295 (acceptJob) | Self-acceptance check is address-based only (case-insensitive comparison). No cryptographic proof that the reviewer is the claimed address. | Require signature verification from reviewer address before accepting. |
| **LOW** | accessService.ts:58-87 | `requireAgentAuthorization()` checks fingerprint and role but doesn't verify the fingerprint was generated by the claimed account. | Add fingerprint derivation verification in authorization check. |
| **INFO** | jobService.ts:298-302 | ReviewerQueue assignment is soft-enforced (logs warning only). Collusion possible if reviewers bypass assignment. | Make ReviewerQueue check hard enforcement or add monitoring/alerting for bypasses. |

## Detailed Analysis

### Agent Impersonation Risk

**Location:** jobService.ts:206-275

The `submitJob()` function accepts `accountAddress` and `agentFingerprint` as parameters but never cryptographically verifies that the fingerprint belongs to the claimed address. An attacker can:

1. Claim a job as agent A (valid address)
2. Submit work with `agentFingerprint` of agent B
3. The submission is recorded with agent B's fingerprint
4. Agent B receives reputation credit for work they didn't do

**Root cause:** No call to verify `computeAgentFingerprint(agentType, agentVersion, accountAddress) === agentFingerprint` at submission time.

### GitHub Token Security Issues

**Location:** githubService.ts:158-186

1. **Plaintext storage:** Tokens stored in `Map<string, { token: string; ... }>` with no encryption
2. **No token validation:** No check if token is revoked or expired during session retrieval
3. **No scope validation:** Token scopes not validated before use
4. **SSRF risk:** `getRepoContext()` accepts unvalidated `fullName` parameter that could include malicious URLs

### Self-Acceptance Prevention

**Location:** jobService.ts:287-295

The check exists and prevents reviewer == worker and poster == reviewer:
```typescript
if (reviewerId.toLowerCase() === (job.activeClaimWorkerId ?? '').toLowerCase()) {
  throw httpError('Reviewer cannot be the same as the worker', 403, 'SELF_ACCEPTANCE_FORBIDDEN');
}
```

However, this is only address-based. Without signature verification, a compromised session could allow bypass.

### Authorization Coverage

All acceptance/rejection routes require session authentication via `requireSessionAccountAddress()` (accessService.ts:18-22), but agent-specific authorization checks are only enforced in `requireAgentAuthorization()` which may not be called on all critical paths.
